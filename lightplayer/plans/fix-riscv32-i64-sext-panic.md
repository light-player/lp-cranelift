# Fix RISC-V32 I64 sext Helper Panic

## Problem Summary

The fixed-point conversion pass generates I64 intermediate operations for fixed-point arithmetic. When lowering these operations on RISC-V32, the `sext` helper function panics because it tries to extract a single register from an I64 value, which requires 2 registers on RV32.

## Root Cause

1. **Fixed-point conversion generates I64 operations**:

   - `sextend` I32 → I64 (2 registers on RV32)
   - `imul` I64 × I64 (2 registers each, result is 2 registers)
   - `sshr` I64 >> 16 (2 registers)
   - `ireduce` I64 → I32 (takes 2 registers, returns 1 register)

2. **Panic location**:

   - `put_in_reg` at line 119 of `cranelift/codegen/src/machinst/isle.rs`
   - Called from `constructor_put_in_xreg` in generated ISLE code
   - The `sext` helper function (`cranelift/codegen/src/isa/riscv32/inst.isle:2148`) calls `put_in_xreg` which internally calls `put_in_reg(val).only_reg().unwrap()`
   - When `val` is an I64 value (2 registers), `only_reg()` returns `None` and `unwrap()` panics

3. **Missing rule**: The `sext` helper only has rules for values that fit in 32 bits (`fits_in_32`). When an I64 value is passed to `sext` (which can happen in the `sextend` rule at line 1326 of `lower.isle`), it falls through to a default rule that calls `put_in_xreg`, causing the panic.

## Backtrace Analysis

```
   5: put_in_reg at cranelift/codegen/src/machinst/isle.rs:119:46
   6: constructor_put_in_xreg at isle_riscv32.rs:3229:14
   7: constructor_lower at isle_riscv32.rs:9418:47
```

The panic occurs when lowering an instruction that uses `sext` on an I64 value.

## Solution

Add a rule to the `sext` helper function to handle I64 values by extracting the low 32-bit register. On RV32, I64 values are represented as a pair of I32 registers, and sign-extending an I64 value means we need to extract the low register (which already contains the sign-extended value).

## Implementation Steps

1. **Add I64 rule to `sext` helper**:

   - File: `cranelift/codegen/src/isa/riscv32/inst.isle`
   - Location: After the existing `sext` rules (around line 2170)
   - Add rule: `(rule (sext val @ (value_type $I64)) (value_regs_get val 0))`
   - This extracts the low 32-bit register from the I64 value pair

2. **Add ISLE filetests for I64 operations**:

   - File: `cranelift/filetests/filetests/isa/riscv32/i64-sext.clif` (new file)
   - Test cases to add:

     - `sext` helper called on I64 value in various contexts
     - `ireduce.i32` from I64 (ensure it works correctly)
     - Fixed-point multiplication pattern: `sextend.i64` → `imul` → `sshr` → `ireduce.i32`
     - Edge cases: negative I64 values, zero, maximum values

3. **Verify the fix**:

## Notes

- The `sext` helper is used by the `sextend` lowering rule, which is why I64 values can end up being passed to it
- On RV32, I64 values are always represented as register pairs, so extracting the low register is the correct way to get the sign-extended value
- This fix is minimal and targeted - it only adds the missing case for I64 values
- The filetests ensure we have coverage for the specific pattern that was failing (fixed-point multiplication with I64 intermediates)
